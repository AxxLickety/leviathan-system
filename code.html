How the repo is organized

Leviathan is built as a research-first system: synthetic world generation, OOS evaluation, robustness checks, and policy design are kept separate so the logic stays auditable.

Code philosophy

1. Separate signal from policy

The regime signal and the overlay activation policy are treated as different questions, so signal validity is not confused with deployment style.

2. Prefer parallel experiments

New worlds and new hypotheses live in new scripts and output folders instead of mutating the original pipeline in place.

3. Keep the story falsifiable

DTI was repeatedly stress-tested and removed from the core model once it failed to add stable incremental value.

What survives in the codebase

The final defensible core is intentionally small:

Signal: adverse macro regime Action: reduce exposure Default policy: 50% de-risk after 2 consecutive adverse quarters

Everything else in the repo shows how that conclusion was earned: through OOS testing, failed branches, harsher synthetic worlds, and policy trade-off analysis.

This page is meant as a code walkthrough for readers who want to understand how Leviathan is built, not just what the final model claims.
